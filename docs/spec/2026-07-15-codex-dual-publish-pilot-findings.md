# Codex Dual-Publish — Pilot Findings & Implementation Learnings

**Status:** Rollout **PAUSED at PR 5a** (2026-07-15).
**Design spec:** [`2026-07-08-codex-dual-publish.md`](./2026-07-08-codex-dual-publish.md)
**Why paused:** The PR 5a pilot showed that Codex does **not** actually spawn a
plugin subagent from a router skill under `codex exec` — even at Ultra
intelligence level. The router degrades gracefully to inline guidance, so the
design is not broken, but the "proactive delegation" premise behind the
`using-*` router skills (spec decisions 4/9) does not hold in the headless
harness that was tested. The maintainer chose to pause and rethink the
agent-delegation approach before mirroring it across the other six plugins.

This document captures what was verified so a future redo starts from evidence
instead of re-deriving it. The 5a implementation branch was torn down; the
essential pieces are reproduced here as reference.

Versions under test: **codex-cli 0.144.3**, **Claude Code 2.1.210**.

---

## 1. What PR 5a built (torn down; reproduce from here)

A "vertical slice" proving the whole agent-distribution design on one plugin
(`marketer`) before fanning out. Four pieces:

1. **`plugins/marketer/agents/marketer.toml`** — hand-maintained Codex source of
   truth, sibling of `marketer.md`. Mapping (spec §Agents):
   - frontmatter `name` → `name`
   - frontmatter `description` → `description`
   - markdown body → `developer_instructions` (verbatim)
   - `tools` and `model` **omitted** (`model: opus` is not a Codex slug; tools
     are not portable — Codex uses `sandbox_mode`)
   - First lines carry a marker comment `solopreneur-managed-agent` so the
     bootstrap skill can distinguish a managed copy from a hand-authored one.
   - Body is wrapped in a TOML **literal** string (`'''…'''`) — no escape
     processing, so backticks / `$VARS` survive. (Guard: the body must not
     itself contain `'''`; all six agent bodies were checked — none do.)

2. **`.codex/agents/marketer.toml`** — generated by
   `scripts/generate-codex-manifests.sh` (surface 3), a **verbatim copy** of the
   source TOML. The generator does not transform the TOML or inject the marker —
   the marker must live in the hand-maintained source.

3. **`plugins/solopreneur/skills/codex-agents-bootstrap/`** — a skill that
   installs plugin agent TOMLs into the Codex user agents directory (see §3),
   because Codex plugins cannot register subagents at install time.

4. **`plugins/marketer/skills/using-marketer/`** — an advisory router skill:
   "dispatch the `marketer` agent for substantial work; invoke `marketer:*`
   skills directly for a quick lookup." No embedded skill list.

`validate-codex.sh` passed (structure gate + 7-plugin install smoke). CI on the
PR (`validate-codex.yml`, CodeRabbit) was green.

## 2. Verified facts — install & discovery (Codex)

Confirmed empirically against a throwaway `CODEX_HOME` (pure file ops, no auth):

- **Install:** `codex plugin marketplace add <repo-root>` reads
  `.agents/plugins/marketplace.json` (marketplace name `solopreneur`); then
  `codex plugin add marketer@solopreneur`.
- **Cache layout:** an installed plugin's files live at
  `$CODEX_HOME/plugins/cache/<marketplaceName>/<plugin>/<version>/`. The version
  segment may be semver (`0.0.8`) or a hash. **The cache preserves the plugin's
  `agents/` directory** — both `marketer.md` and `marketer.toml` land there. This
  is the load-bearing assumption for the bootstrap skill, and it held.
- **Enumerating installed plugins:** `codex plugin list --json` returns
  `{ "installed": [ { name, marketplaceName, version, enabled, … } ], "available": [] }`.
  That (name, marketplaceName, version) triple reconstructs the cache path above.

## Verified facts — agent registration (Codex)

Per the official docs (learn.chatgpt.com/docs/agent-configuration/subagents) and
confirmed in a diagnostic run:

- User subagents are **auto-discovered** from `~/.codex/agents/*.toml` (personal)
  or `.codex/agents/*.toml` (project). No `config.toml` registration needed.
- Required TOML fields: `name`, `description`, `developer_instructions`. Optional:
  `model`, `model_reasoning_effort`, `sandbox_mode`, `mcp_servers`,
  `nickname_candidates`, `skills.config`.
- A diagnostic prompt confirmed Codex **sees** the bootstrapped agent: asked to
  list delegatable agents, the model returned `marketer`. So registration works;
  discovery is not the problem.

## Verified facts — Claude side

- **Fixture test:** a plugin carrying both `agents/foo.md` and `agents/foo.toml`
  (shared basename, to force any collision) installed into a throwaway
  `CLAUDE_CONFIG_DIR`. `claude plugin details` reported **`Agents (1) foo`** — only
  the `.md`; no phantom agent, no error. `claude plugin validate` passed. So a
  `.toml` sibling is inert on Claude Code (spec decision 4 holds).

## 3. Bootstrap skill — mechanics that worked

The skill copies each installed plugin's `agents/*.toml` from the cache into the
Codex user agents dir, guarding hand-authored files. Verified end-to-end
(install / marker-guard / orphan-detection) against a real install. Core logic:

- Resolve `CODEX_ROOT="${CODEX_HOME:-$HOME/.codex}"`; agents dir `$CODEX_ROOT/agents`,
  cache `$CODEX_ROOT/plugins/cache`.
- Enumerate sources: `codex plugin list --json` → for each installed
  `<marketplaceName>/<name>/<version>`, glob `<cache>/…/agents/*.toml`. Fallback
  when `jq`/JSON is unavailable: glob `<cache>/*/*/*/agents/*.toml`.
- Copy each source that carries the `solopreneur-managed-agent` marker. Before
  overwriting a same-named file in the agents dir, require **that file** to also
  carry the marker; otherwise skip it (never clobber a hand-authored agent).
- Orphan = a marker-carrying file in the agents dir with no matching installed
  plugin. Report; delete only on user confirmation.

This part is sound and reusable as-is if the rollout resumes.

## 4. Pilot findings — agent spawning (the decisive result)

**Question:** does a `using-*` router skill cause Codex to spawn the plugin's
subagent? Tested in a fully-isolated Codex home (throwaway `HOME` + `CODEX_HOME`,
copied `auth.json`) so only the marketer plugin's skills were in scope.

**Result: no real subagent thread was ever spawned under `codex exec`.**

| Prompt | Level | Read `using-marketer`? | Real spawn? |
| --- | --- | --- | --- |
| "Use the marketer subagent to write positioning…" (explicit) | default | yes | no |
| "Help me with positioning, a name, launch copy…" (proactive) | default | yes | no |
| same explicit prompt | **Ultra** (`gpt-5.6-sol`, `reasoning_effort=ultra`) | yes | no |
| same proactive prompt | **Ultra** | yes | no |

Across all four runs plus a diagnostic, the only agent-collaboration tool the
model ever invoked was `collab_tool_call` with `tool: "wait"` and **empty
`receiver_thread_ids`** — i.e. it waited on nothing. The model *narrated*
delegating to a "marketer subagent" but produced the output itself, inline. No
`spawn`/`create`/`send` collaboration call ever appeared.

**Why.** The docs state the model "does not proactively spawn subagents based on
descriptions except at Ultra level." Our tests show that even at Ultra,
`codex exec` (headless) does not produce a real spawn — the strong hypothesis is
that real subagent threads are an **interactive TUI / app** capability that is
not wired into `codex exec`. This was **not** tested in the interactive TUI (hard
to drive headlessly); the handoff scoped the pilot to `codex exec`, so the
finding is scoped there too.

**What *did* work:** the router is discovered and followed. In every run the
model read `using-marketer` and pulled in the right marketer skills
(positioning, naming, humanly), producing genuine marketing output. So the
router delivers value as **inline guidance**, and the agent stays discoverable
for explicit / interactive use.

## 5. Implications for a redo

- The `using-*` routers are **not wasted** — they work as inline guidance and are
  the spec's own pre-decided fallback (spec §Rollout, pilot-gate fail path). If
  the rollout resumes with delegation de-emphasized, the routers barely change.
- Before committing to "proactive delegation," **re-test in the interactive Codex
  TUI / app**, not just `codex exec`. If spawning only works interactively, the
  design should either (a) target that harness explicitly, or (b) drop
  agent-spawn as a premise and lean fully on inline routing + direct skills.
- The bootstrap skill, the TOML mapping, the generator wiring, and the cache/
  discovery facts above are all verified and reusable — a redo can start at the
  router-wording / delegation question rather than the plumbing.

## 6. Gotchas encountered

- **Global-skills contamination:** on this machine, a large merged skills
  collection at `/Users/Hana/Agents/skills/_merged/` bled into a `CODEX_HOME`-only
  isolated home (the model read gtm's SKILL.md from there, and Codex reported the
  "2% skills context budget" truncation because hundreds of skills were in scope).
  A real plugin user has none of this. **Full isolation requires overriding `HOME`
  too**, not just `CODEX_HOME` — otherwise `~`-anchored global skills leak in.
- **Authenticated throwaway home:** the spawn test needs a logged-in Codex home.
  Copying just `auth.json` into a fresh `HOME/.codex` yields an authenticated
  isolated home without polluting the real one (`codex login status` → "Logged in
  using ChatGPT"). Clean it up afterwards — it holds a real token.
- **Intelligence level is a hidden variable:** copying only `auth.json` (not
  `config.toml`) runs Codex at a default level, not the maintainer's usual Ultra.
  Replicate the level explicitly (`-c model=… -c model_reasoning_effort=ultra`)
  when a behavior might depend on it — proactive delegation does.

## 7. Open questions for the redo

1. Does real subagent spawning happen in the **interactive Codex TUI / app** (vs
   `codex exec`)? This is the crux — it decides whether "delegation" is viable at
   all for plugin users.
2. If spawning is interactive-only, is inline routing (router + `marketer:*`
   direct skills) enough to call the Codex parity "done," or is a spawned agent a
   hard requirement?
3. Does the "2% skills context budget" truncation degrade routing quality once a
   plugin ships many skills? Worth measuring on a clean install with only the
   plugin present.
